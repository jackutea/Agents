# Bootstrap Agent

## 目的

该 skill 用于两类任务：

1. 创建新的 agent。
2. 修改已有 agent。

目标是产出结构清晰、约束明确、可直接使用的 agent 文件，并确保 agent 的职责、触发条件、边界和输出方式一致且可执行。

## 适用场景

在以下情况使用本 skill：

- 用户要求创建一个新的 agent。
- 用户要求修改、重写、收敛或扩展一个已有 agent。
- 用户要求完善 agent 的说明、工作流、限制条件、使用场景或输出规范。
- 用户要求梳理 agent 的职责边界，避免与其他 agent 重叠。

在以下情况不要使用本 skill：

- 只是修改普通代码文件，而不是 agent 文件。
- 只是解释 agent 概念，没有要求落地到文件。
- 只是修复运行时代码错误，且问题不在 agent 定义本身。

## 强制约束

### 1. 禁止参考项目内已有文档

执行本 skill 时，不得把当前工作区中已有的文档作为参考来源，包括但不限于：

- 已有的 agent 文件
- 已有的 skill 文件
- 已有的 instructions 文件
- 已有的 prompt 文件
- 其他说明性 Markdown 文档

允许使用的信息来源只有：

- 用户当前明确提供的要求
- 当前正在编辑的目标 agent 文件本身
- 通用的 agent 设计原则
- 官方能力或平台约束

如果缺少信息，不要通过搜索项目文档补足；应直接向用户提问。

### 2. 每个 agent 必须包含三块核心内容

凡是通过本 skill 创建或修改的 agent，正文都必须明确包含以下三块内容，缺一不可：

- 接收的 Input
- 处理的事项
- 输出的 Output

这三块内容必须写成用户可读、可执行、可检查的规则，不能只在描述中隐含表达。

如果当前 agent 草稿缺少其中任意一块，必须先补齐，再继续完善其他内容。

### 3. Header 改动前必须先询问用户

只要任务涉及 agent header 或 frontmatter 的创建、修改、补全、删减、重命名，都必须先询问用户并等待答复，然后才能继续编辑。

这里的 header 包括但不限于：

- `name`
- `description`
- `model`
- `tools`
- `version`
- YAML frontmatter 中的任意字段

禁止跳过确认直接改 header，即使改动看起来显而易见也不可以。

### 4. 涉及 skill 时必须先列出并让用户选择

默认解释规则：当用户提及 agent 时，视为也同时提及该 agent 对应的 skill。

这意味着：

- 只要用户提出创建 agent，就必须同步评估是否需要新增对应 skill
- 只要用户提出修改或维护 agent，就必须同步评估是否需要修改对应 skill
- 不能把 skill 视为只有用户单独点名时才需要考虑的附属项

当创建 agent 或维护 agent 的过程中，发现还需要新增 skill、修改 skill、补全 skill、重构 skill 时，不能直接处理这些 skill。

必须先完成以下动作：

- 明确列出本次涉及哪些 skill
- 说明每个 skill 是新增还是修改
- 说明每个 skill 解决什么问题
- 让用户明确选择本次要先处理哪些 skill

在用户完成选择前，不得进入对应 skill 的创建或修改。

## 执行流程

### 第一步：确认任务类型

先判断当前任务属于哪一种：

- 新建 agent
- 修改 agent 正文
- 修改 agent header
- 同时修改 header 与正文

如果包含 header 改动，先进入“Header 问询流程”，不要直接编辑。

### 第二步：收集最少必要信息

只收集完成当前任务所需的最小信息，优先从用户请求中提取：

- agent 的用途
- 目标用户或调用方
- 能做什么
- 不能做什么
- 触发条件
- 输出格式
- 是否有工具限制
- 是否需要分阶段工作流
- 是否会涉及新增或修改 skill

当用户只提到 agent、未单独提到 skill 时，默认按“同时提到了对应 skill”处理。

如果这些信息不足以写出可靠内容，向用户发出简短且明确的问题，不要自行脑补。

### 第三步：先整理三块核心内容

在进入正式写作前，先明确当前 agent 的三块核心内容：

- 接收什么 Input
- 处理什么事项
- 输出什么 Output

如果用户只描述了用途，没有说明输入或输出，不要跳过；应向用户补问，直到这三块内容都可明确落笔。

### 第四步：判断是否需要问询 header

若出现以下任一情况，必须先问用户：

- 要新建 agent header
- 要改已有 header 的字段值
- 要新增 header 字段
- 要删除 header 字段
- 要调整 header 字段顺序且会影响可读性或约定

推荐提问模板：

1. 这个 agent 的 `name` 是否需要调整？
2. `description` 希望偏向任务描述、能力边界，还是触发词优先？
3. 是否需要声明 `model` 或 `tools` 等额外字段？

只有在用户回复后，才能继续 header 编辑。

### 第五步：判断是否涉及 skill 变更

在本步骤中，先应用默认解释规则：只要用户提到了 agent，就先视为对应 skill 也进入评估范围。

若当前 agent 的创建或维护会连带产生 skill 层面的工作，必须先整理一份 skill 清单。

skill 清单至少应包含：

- skill 名称
- 新增或修改
- 处理目的
- 为什么与当前 agent 任务相关

整理完成后，必须先给用户选择，再继续处理被选中的 skill。

若用户尚未选择：

- 不创建 skill
- 不修改 skill
- 不把 skill 改动混入当前 agent 改动中一起提交

### 第六步：编写或修改正文

在不参考项目现有文档的前提下，围绕当前任务直接编写正文。正文应尽量覆盖：

- 目的
- 适用场景
- 不适用场景
- 接收的 Input
- 处理的事项
- 输出的 Output
- 输入要求
- 执行步骤
- 约束
- 输出要求
- 常见失败点或回退策略

正文要避免空泛原则，优先写成可以直接执行的步骤或判定规则。

### 第七步：自检

完成后至少检查以下内容：

- header 是否满足用户确认结果
- 内容是否直接服务于“创建 agent / 修改 agent”
- 是否写入了明确的边界与限制
- 是否违反“禁止参考项目内已有文档”的约束
- 是否明确包含 Input、事项、Output 三块核心内容
- 是否把“涉及 header 先问用户”写成强制规则
- 若涉及 skill，是否先列出 skill 清单并让用户完成选择
- 描述是否足够具体，能被准确触发

## Header 问询流程

当任务涉及 header 时，使用明确问题向用户确认，不要把多个不相关问题混成一大段说明。

最少确认以下内容：

1. `name` 是否确定。
2. `description` 是否需要调整。
3. 是否需要额外 header 字段。

若用户没有给出明确答案：

- 不要擅自提交 header 改动。
- 可以先整理正文草案。
- 明确标注哪些部分因 header 信息缺失而暂缓。

## 编写原则

- 只做当前任务需要的最小改动，不顺手扩写无关内容。
- 优先修正职责边界不清、流程不可执行、触发描述模糊这三类问题。
- 对每个 agent，都要把 Input、事项、Output 单独写清，不能合并成模糊段落。
- 当用户提及 agent 时，默认同步检查对应 skill，不把 skill 排除在评估之外。
- 若任务会影响 skill，先列出 skill 清单并让用户选定范围，再开始动 skill。
- `description` 要包含触发词和任务意图，避免只有抽象描述。
- 如果 agent 有明确禁区，要写成禁止项，不要只写建议项。
- 如果 agent 需要提问再继续，必须明确写出触发条件和提问时机。

## 输出要求

完成任务时，输出应简洁说明：

- 本次是新建还是修改 agent
- 是否涉及 header
- 如果涉及 header，用户确认了哪些字段
- 是否涉及 skill
- 如果涉及 skill，列出了哪些 skill，用户选择了哪些 skill
- 本次改动解决了什么问题
- 如仍有未决项，明确指出缺失信息

## 质量标准

一个合格的 agent skill 至少满足以下条件：

- 能指导创建 agent
- 能指导修改 agent
- 能强制 agent 具备 Input、事项、Output 三块核心内容
- 能在 header 变更时强制进入问询流程
- 能在涉及 skill 时先列出 skill 清单并要求用户选择
- 不依赖项目内已有文档
- 规则具体，可执行，不含含糊表述
- 产物能直接用于后续编辑工作
